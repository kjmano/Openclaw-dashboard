Analyse pragmatique

Limites d’usage openai-codex/gpt-5.4

Cette page répond à une question simple : combien de tokens peut-on utiliser avant de tomber sur un “API rate limit reached” avec openai-codex/gpt-5.4 dans OpenClaw ?

Réponse courte : il n’existe pas un seul “nombre magique” public. Les limites dépendent du type d’accès, du compte, du tier, du débit de requêtes et parfois d’un quota glissant.

Conclusion principale
Pas de plafond unique public

Le message de rate limit peut être causé par le débit (RPM/TPM), pas seulement par le total accumulé.

Ce que la doc publique confirme
RPM + TPM + usage tier

Les limites OpenAI combinent requêtes/minute, tokens/minute, parfois requêtes/jour ou tokens/jour.

Cas probable si attente très longue
Quota / fenêtre glissante

Attendre plusieurs jours ressemble moins à un petit pic minute qu’à une restriction liée au compte ou au quota.

1) Ce que dit la documentation publique

OpenAI API — rate limits

  • OpenAI documente des limites de type RPM (requests per minute) et TPM (tokens per minute).
  • Les limites varient selon le modèle.
  • Elles dépendent aussi du tier d’usage du compte.
  • La page explique aussi qu’on peut voir les limites exactes dans la console développeur du compte.

Source publique utilisée : OpenAI API docs — “Rate limits”.

Page modèle GPT-5 (référence publique proche)

  • La page publique GPT-5 affiche des limites par tier pour l’API directe.
  • Exemple visible publiquement pour GPT-5 API : Tier 1 = 500 RPM et 500,000 TPM.
  • Mais ça ne prouve pas que le même plafond s’applique tel quel à openai-codex/gpt-5.4 via Codex OAuth dans OpenClaw.

Important : c’est un point de comparaison, pas une preuve de la limite exacte de ton accès.

2) Ce qu’OpenClaw confirme

OpenClaw documente que openai-codex/gpt-5.4 est le mapping utilisé pour l’accès Codex OAuth / ChatGPT-Codex. Donc ton usage ici n’est pas forcément identique à un usage API classique avec clé OpenAI.

En clair : modèle proche, chemin d’accès différent, règles de quota potentiellement différentes.

3) Estimation pour ton cas

Ce qu’on peut estimer

  • Dans la session que j’ai inspectée, on a vu un point de mesure à environ 8.3k tokens input et 52 output.
  • Ça donne un ordre de grandeur d’environ 8.3k à 9k tokens autour du moment où on parlait du rate limit.
  • Ce volume, à lui seul, n’a rien d’énorme.

Ce que ça suggère

  • Si tu as vraiment dû attendre 6 jours, le problème ne ressemble pas à un simple dépassement ponctuel de tokens sur une minute.
  • Le scénario le plus crédible est plutôt un quota de compte, une fenêtre glissante longue, ou une politique de limitation propre à l’accès Codex OAuth.
  • Donc : le blocage semble plus lié à la politique d’accès qu’à un total du style “tu as atteint exactement X tokens”.

4) Ce qu’on peut dire sans mentir

  • Oui, il existe des limites de requêtes et de tokens côté OpenAI.
  • Oui, elles varient selon le modèle et le tier.
  • Non, je n’ai pas de preuve publique fiable d’un plafond unique et exact pour openai-codex/gpt-5.4 dans ton mode d’accès actuel.
  • Non, les ~8.3k–9k tokens observés ne suffisent pas à expliquer à eux seuls une attente de 6 jours.

5) Mon verdict

Le plus honnête est de dire ceci :

Pour openai-codex/gpt-5.4, la limite que tu as rencontrée ne semble pas correspondre à un simple total public de tokens. Avec les éléments visibles, ton incident ressemble davantage à un quota ou à une limitation de service liée au compte / au mode d’accès Codex, possiblement sur une fenêtre longue, qu’à un plafond simple du type “tu as utilisé X tokens et c’est fini”.
